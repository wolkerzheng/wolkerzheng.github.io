Guide

怎么把这个博客长期维护下去

这次我先用“静态优先”的方式，把你现有的 GitHub Pages 站点整理成一个更现代、更清晰的内容入口。 如果你希望后续更新轻松、低成本、可持续，建议把维护节奏和栏目结构先固定下来，再决定是否做第二阶段技术升级。

Weekly

建议的轻量更新节奏

  • 每周补一条短笔记：工具、踩坑、阅读摘录都可以。
  • 每月发一篇稍完整的主题文章，沉淀成可复用内容。
  • 每季度更新一次首页精选文章和技术栈描述。

Content Buckets

后续内容建议分成三类

  • 深度文章：NLP、推荐系统、模型理解、工程专题。
  • 工作笔记：框架使用、数据处理、效率工具、实验经验。
  • 职业记录：项目复盘、成长总结、面试与求职经验。

Keep It Fresh

让博客看起来一直在更新

  • 首页精选文章至少保留一篇近 3 个月内容。
  • 新增一个“最近在做什么”的短内容栏目，哪怕每次只写 100 字。
  • 简历页、精选文章、维护页三者一起更新，网站就会显得是活的。

Current Repo

你现在这套仓库的现实情况

当前仓库保存的是已经生成好的静态页面，而不是 Hexo 或其他框架的源码仓库。 这意味着它部署很直接，但后续手工改博客会比较笨重，不适合长期高频写作。

Recommended Phase 2

下一阶段更推荐的技术栈

如果你准备把它持续运营下去，我建议后面单独做一版源码仓库，再自动部署到这个 GitHub Pages 仓库。

内容层

推荐 Astro + MDX。静态输出很适合博客，文章用 Markdown/MDX 写，维护成本低，加载也快。

工程层

推荐 TypeScript + Content Collections。文章元数据、标签、栏目会更清晰，后面做归档和专题页也方便。

部署层

推荐 GitHub Actions 自动构建并发布到 github.io 仓库，这样以后更新博客就是写文然后 push。

运营层

可以加一个 Now / Notes 轻笔记栏目，让网站不必等大文章写完才有更新。

Quick Checklist

你后面每次更新时，最小动作清单

  1. 更新一篇文章，或者补一条简短近况。
  2. 如果内容值得展示，把首页“精选内容”替换一条。
  3. 如果工作经历有变化，同步替换简历 PDF。
  4. 每隔一段时间检查一下首页的技术栈表述有没有落后。

我也在仓库根目录补了一份 维护说明， 方便你以后直接在代码仓库里查看。

先把更新节奏跑起来，再决定要不要做第二阶段的源码化改造。
