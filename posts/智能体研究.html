AI 智能体 sandbox 2026-04-17

智能体（Agent）

Agent 能力边界 = Sandbox 设计能力。 OpenClaw、Manus、Claude Code、Dify，其实代表了 4种完全不同的 sandbox 架构路线。 一、先把 Sandbox 讲透（核心认知） 在 Agent 里，sandbox 本质是： 给 LLM 一个“可控的行动世界” 它解决 3 个核心问题： 1️⃣ 安全边界（Security Boundary）

2️⃣ 执行环境（Execution Runtime）

3️⃣ 状态容器（State Container）

👉 简单一句话总结： Sandbox = Agent 的“操作系统”

  • 防止模型执行危险操作（删库 / 发请求 / 乱调用API）
  • 权限隔离（文件 / 网络 / 系统）
  • 代码运行（Python / Bash / Node）
  • 工具调用（API / DB / 浏览器）
  • 文件系统（workspace）
  • 运行上下文（memory / artifacts）

--- 二、四大系统 Sandbox 架构对比（重点） 我给你直接上一个“架构级对比”： 暂时无法在飞书文档外展示此内容

--- 三、逐个拆解（重点差异）

--- 1️⃣ OpenClaw：“本地可控操作系统型 Sandbox” 👉 核心设计：

- 文件读写 - Shell - Python执行

👉 本质： 一个“可编程的自动化执行环境” 👉 特点：

👉 优点：

👉 缺点：

  • 本地运行（Docker / CLI）
  • 有完整 workspace
  • 支持：
  • Memory + Tool 强绑定
  • sandbox ≈ 本地 Linux 子系统
  • 强状态（文件持续存在）
  • 可做复杂任务链（编译、爬虫、分析）
  • 可控性极强
  • 可扩展（你可以自己加工具）
  • 安全要自己兜底
  • 运维复杂

--- 2️⃣ Manus：“云端虚拟人 Sandbox” 👉 核心设计：

- 打开网页 - 写代码 - 下载文件 👉 本质： 模拟“一个人在电脑前工作” 👉 特点：

👉 优点：

👉 缺点：

  • 云端 VM / Browser / Tools
  • Agent 像“真人”一样操作：
  • sandbox ≈ 云电脑
  • 多模态（浏览器 + 文件 + API）
  • 强 autonomy（自主决策）
  • 最接近 AGI 形态
  • 通用性极强
  • 成本高
  • 可控性弱（容易乱跑）

--- 3️⃣ Claude Code：“IDE内嵌 Sandbox” 👉 核心设计：

- 读写文件 - 修改代码 - 运行测试 👉 本质： 一个“安全的代码操作代理环境” 👉 特点：

- 不允许随便访问系统 - 不允许任意网络 👉 优点：

👉 缺点：

  • 嵌入开发环境（类似 VSCode）
  • 操作代码仓库：
  • sandbox ≈ Git repo + limited runtime
  • 强约束：
  • 安全性高
  • 对开发者友好
  • 不通用（偏 coding）
  • 工具受限

--- 4️⃣ Dify：“函数调用型 Sandbox（最轻）” 👉 核心设计：

- Tool（API） - Workflow 👉 本质： 把 Sandbox 抽象成“函数调用” 👉 特点：

👉 优点：

👉 缺点：

  • 没有真正 sandbox
  • 只有：
  • sandbox ≈ API 调用集合
  • 无文件系统
  • 无真实执行环境
  • 企业级稳定
  • 安全（因为啥都不给）
  • 能力弱
  • 不支持复杂任务（如多步代码执行）

- 企业 - 金融 - SaaS 👉 本质： Agent = 工作流引擎 + API

--- 六、一个工程落地方案

👉 最优架构其实是： ✅ “双 Sandbox 架构” Dify（控制层） ↓ OpenClaw（执行层） ↓ 云 sandbox（扩展，如 Manus） 👉 各自分工：

  • Dify
  • OpenClaw
  • 多Agent系统
  • Dify：流程编排 / 权限控制
  • OpenClaw：执行复杂任务
  • 云 sandbox：高阶任务（浏览器/多模态）

--- 七、一句话总结 Sandbox 是 Agent 的“手和脚”执行的环境，模型只负责“大脑”。
